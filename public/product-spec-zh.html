总述

SlashCard 是什么、为谁服务、为什么存在。一页纸读懂。

一句话定义

SlashCard 是对标 todey.xyz 等通用加密卡聚合器的深度替代方案。它不是一张静态对比表，而是一个活的情报平台 —— 汇集真实用户评价、实时问题报告、运营整理的活动/费率情报，以及社区验证层。

我们解决的问题

目前加密卡用户获取决策信息的渠道非常碎片化：

  • Twitter/X —— 嘈杂、无结构、立场化
  • Telegram 群 —— 消息被冲走、每张卡一个群相互孤立
  • 聚合网站（todey.xyz、cryptonomist） —— 一次性访问、没有社区信号、没有真实用户体验
  • 官方卡片网站 —— 营销宣传，没有事故数据

没有一个规范的落地页能回答这个问题："X 这张卡现在到底怎么样？"

MVP 核心功能

🎴 卡片目录

预收录 15 张加密卡，带 CSS 渲染的信用卡视觉、评分与元数据。可按类型筛选（CEX / DeFi / Prepaid / Debit）。

⭐️ 用户评价

1-5 星评分 + 文本评价 + 可选截图。无需登录，10 字符下限。

/ Slash 报告

快速事件报告（冻卡、延迟、KYC 问题）。驱动 Live Status 状态指示器。

📡 Intelligence Feed

运营整理的情报：活动、费率变动、功能更新、技巧。配合社区 confirm/deny。

🚦 Live 状态指示器

绿/黄/红三级状态，基于 7 天滚动 slash 数量计算。

📤 可分享状态卡片

提交后自动生成 PNG 状态卡片。一键分享到 X。

目标用户

持有 1-3 张加密卡、积极使用 USDC/USDT 消费的 crypto-native 用户，主要在亚太和欧洲地区。年龄 22-40。活跃于 Crypto Twitter。

北极星指标

North Star

当有人问 "X 这张卡值得办吗？" 或 "X 卡是不是出事了？" —— 别人扔过来的链接是 SlashCard。

产品愿景与核心论点

我们相对竞品的定位，以及产品的核心赌注。

频率论点

一次性聚合器 和 日活 App 之间是一个光谱。大多数加密卡用户不需要每日打开，但也不止打开一次。

回访频率光谱
  一次性                            每周                            每日
  │                                     │                               │
  todey.xyz ─────────────── SlashCard ────────────── Twitter/X
  （选卡时开一次）          （事件 + 活动 + 充值前                 （每天刷）
                             都会回来看）

用户为什么回来

  1. 充值之前 —— 查看当前费率、有无活动、系统有无问题。每周反射性检查。
  2. X 上看到卡圈新闻 —— 点进 SlashCard 看全貌。事件驱动。
  3. 被烧了 —— 提交 slash 报告警告他人。情绪驱动，强传播性。
  4. 朋友问起 —— 丢一个链接代替打字。转移信息负担。

用户画像

三个核心画像驱动所有设计决策。

P1

选卡人

画像：即将开第一张加密卡。有 ETH/USDC，想线下消费。

目标：选对卡，不踩坑。

痛点：选项太多，不相信卡方营销，想看真实体验。

访问：研究阶段 2-3 次。之后很少。

P2

多卡优化者

画像：持 2-3 张加密卡。根据场景切换使用。

目标：每笔交易价值最大化。避免手续费，抓住活动。

痛点：卡方悄悄改费率。活动埋在 TG 里。

访问：每周充值前。核心回访画像。

P3

被烧用户

画像：刚刚出事。卡被冻、钱卡住、客服失联。

目标：发泄、警告他人、看看自己是不是一个人。

痛点：愤怒。想立刻行动。

访问：事故期间一次。高传播性输出。

V1 画像优先级

P2（多卡优化者）是主要留存目标。P3（被烧用户）是主要获客驱动（通过可分享状态卡）。P1（选卡人）是次要获客目标（通过 SEO + 自然发现）。

用户任务（JTBD）

当用户"雇佣" SlashCard 时，他们想完成的任务。

任务描述 画像 主要功能
当我在选加密卡的时候，我想看到真实用户的体验 —— 这样我就不会在烂卡上浪费钱。 P1 评价 + 评分
当我准备充值的时候，我想查我的卡是否正常、有没有活动 —— 这样我就能价值最大化。 P2 状态指示器 + Intel Feed
当我在 X 上看到有人说卡出事，我想确认是不是普遍问题 —— 这样我知道该不该担心。 P2 Slash 时间轴 + Live Feed
当我的卡刚被冻了，我想告诉全世界 —— 这样别人不会踩坑，我也不再孤单。 P3 Slash 报告 + 分享到 X
当朋友问我推荐哪张卡，我想甩一个链接过去 —— 这样我就不用每次都重复解释。 P2 卡片详情 URL（SEO）

      │    (Rain)    │    │  (RedotPay)  │    │    (...)     │
      └───┬──────────┘    └──────────────┘    └──────────────┘
          │
          ├──► 写评价
          │
          └──► Slash 报告 ──► 分享成功页（带 X 按钮）

持续元素

  • Header：置顶、毛玻璃。Logo（→ 首页）、"Crypto Card Reviews" 标签、"Live" 脉冲指示器。
  • Footer：静态、每页底部。独立性声明（与任何卡方无关联）。

评价表单规范

低门槛评价提交。从原 6 维度 spec 简化而来。

简化决策

为什么砍掉了 6 维度评分

原 spec 有 6 个独立的 1-5 评分（充值速度、费率、链支持、冻卡风险、客服、KYC）。对零评价的冷启动产品来说，让用户填 6 个滑块是转化杀手。MVP 只用 1 个总评分。多维度评分在 V2 有数据量时再回归。

验证与错误处理

  • 错误显示在提交按钮上方的红色 callout 里
  • 一次一个错误（first-failure 获胜），不罗列
  • 错误文案直接：用 "Please select a rating" 而不是 "A valid rating is required"

Confirm / Deny 机制

V1 最重要的 UX 决策

Confirm/Deny 是 V1 中用户对情报的唯一贡献方式。因为只需一次点击，它能以约 100:1 的比例把被动读者转化为参与者（对比任何 UGC 提交）。

按钮样式
[✓ N] 绿色描边 · [✗ N] 红色描边。紧凑（8×24px padding）。tabular-nums。
单次投票
投票后两个按钮都禁用。选中按钮填充，另一个淡化（opacity 0.4）。
持久化
V1：仅内存（刷新重置）。V2：localStorage + 服务端去重。
审核
如果 deny > confirm + 3，标记待运营审查。deny > 10 自动隐藏。[V1.1]

Feed 筛选

Feed 顶部的 tab 控件：All · Intel · Reports

  • All：intel + slash + reviews，按时间
  • Intel：仅运营发布的情报（promo/fee/feature/tip）
  • Reports：仅 slash 报告

Feed 排序

V1：严格时间顺序（最新在前）。

V2 考虑：按 confirm - deny 加权（质量信号）。置顶高优先级条目（新活动、红色状态卡告警）24h。

运营 / 管理

  • 💡 技巧

用户

用户贡献的内容

  • ⭐️ 评价（完整提交）
  • / Slash 报告（完整提交）
  • ✓/✗ Confirm/Deny（一次点击）

动机：评价 = 无门槛的情绪输出。报告 = 愤怒。Confirm = 近零成本参与。

V1 为什么不接受用户提交情报

冷启动现实

我们分析过每种内容类型的用户动机。正向情报（活动、技巧）的创作需要声誉、积分或社区归属感激励 —— V1 都没有。让匿名用户提交情报 = 空情报流。

改为：V1 团队承担供给。用户承担验证（confirm/deny）。当用户量达到一定规模（~1000 DAU），再开放用户提交情报，配合适度审核。

运营手册（内部团队）

  1. 每日：扫 X + TG 群找卡片新闻。带 source link 发布。
  2. 每周：至少发 1 条 tip。如有 deal，发 4-8 条 promo。
  3. 每月：审计所有卡片的费率变化列表。对比官方费率表。
  4. 事件驱动：当某卡事件激增时，写一条 summary 情报为活动加上下文。

状态指示器算法

绿 / 黄 / 红 如何决定。

规则

  绿 (Normal)         recent_slash_count ≤ 5
  黄 (Issues)         5 < recent_slash_count ≤ 15
  红 (Critical)       recent_slash_count > 15

  recent_slash_count = slash_reports WHERE created_at > now() - 7 days

为什么用 7 天滑动窗口

用固定"全部时间"的阈值会导致状态永久不变。滑动窗口让状态自愈 —— 老报告过期后状态会恢复。如果某卡 2 个月前有尖峰但现在正常，会恢复到绿色。

为什么用绝对阈值（不是比例）

基于比例的阈值（如"报告/用户数"）需要知道每卡用户基数 —— 我们不知道。绝对阈值简单，可以根据预期量校准。等有流量数据后再按卡片热度调整。

视觉处理

绿
Badge "Normal"。脉冲点。卡片无光晕。平静、不打扰。
黄
Badge "Issues (N)"。脉冲琥珀点。详情页微黄光晕。首页脉冲警示条。
红
Badge "Critical (N)"。脉冲红点。详情页强红光晕。优先警示条。Trending 排序置顶。

反作弊考虑

风险：刷榜 / 竞品攻击

V1 开放匿名提交。竞品可以提交 20 条 slash 报告把对手搞成红色。

缓解（V1.1）：按 IP + 浏览器指纹限流（每卡 24h 1 条）。要求 confirm/deny 投票加入状态计算（低 confirm 的报告权重低）。

文案与语气

语气准则与微文案约定。

语气原则

✓ 做

  • 直接。"Please select a rating" 不是 "A valid rating must be selected"
  • 对卡片中立。不为任何卡背书或攻击。
  • 对问题事实化。"3 reports this week" 不是 "灾难正在展开"
  • 除专有名词外小写。
  • 用 em-dash（—）停顿，不用冒号。

✕ 不做

  • 说教。（删掉了 "No shills. No sponsors."）
  • 营销套话。不说 "revolutionary" "game-changing"。
  • 正文用 emoji。只在特定 UI 位置（情报类型图标）用。
  • 道歉。"Report submitted" 不是 "Sorry for the trouble!"
  • 感叹号。永远不用。

微文案约定

场景我们的文案为什么不用替代方案
空评价"No reviews yet"不用 "Be the first!" —— 避免鼓动
分享按钮"Share to X"不用 "Tweet" —— X 已改名
Slash 表单标题"/ Slash Report"slash 符号本身就是品牌元素
状态"Normal / Issues / Critical"不用 "OK / Warning / Down" —— 更有语义
报告成功"Help spread the word so others are warned"把分享定义为利他，不是自我推广
评价提示"Top-up speed? Freeze issues? Support quality?"引导写作者，不规定结构

增长闭环

SlashCard 如何自我增长，不靠付费获客。

病毒机制

          ┌─────────────────────────────┐
          │  用户经历卡片事故            │
          └────────────┬────────────────┘
                       ▼
          ┌─────────────────────────────┐

          │ 或通过 SEO 落地              │
          └────────────┬────────────────┘
                       ▼

          │ 读到 slash 报告，发现自己   │
          │ 不孤单                       │
          └────────────┬────────────────┘
                       ▼
          ┌─────────────────────────────┐
          │ 提交自己的 slash 报告        │
          └────────────┬────────────────┘
                       ▼

          │ 传到关注者                   │
          └────────────┬────────────────┘
                       │
                       ◄──── (闭环)

病毒系数估算

每次 slash 提交 → 约 25% 点 "Share to X"（参考类似金融事件 App）→ 平均推文触达约 300 关注者 → 推文到 SlashCard 点击率约 2% → 每次提交约带来 0.15 次新访问。

按这个速率，光靠 slash 报告不足以规模化增长。计划补充：

  • 运营自动发推：当某卡进入 RED 状态，运营手动发状态卡片图（不需要改 UI，只是 playbook）
  • 周报：邮件订阅者收到 "This week in crypto cards" —— 高信号低成本留存
  • 可嵌入状态组件：卡方 / 博主可嵌入 "X 卡当前状态" badge → 反向链接（V2）

待讨论问题与 V2 路线图

需要讨论和接下来要做的事。

待讨论设计问题

  1. "Trending" 排序语义：应该按报告最多（现状）还是按最近活动最多（无论正负）？即有大量新活动的卡也该 "trend"？
  2. 警示条选择：目前显示第一个红/黄卡。应该是最新事件？还是最高计数？还是用户可配？
  3. 评价表单字数门槛：10 字符下限合适吗？太高杀转化，太低招垃圾。V1.1 监控 spam 率。
  4. Feed 内情报排序：纯时间顺序还是按 confirm 加权？需要置顶机制吗？
  5. 详情页 CTA 优先级：目前 Review 是主（靛青）。对有问题的卡正确吗？状态红时要不要切换成 Slash 为主？

V1.1（上线后 1-2 周）

  • Supabase 后端（替换 mock 数据）
  • 提交限流（每 IP 每卡每 24h 1 次）
  • 原生 alert() 换成 toast 组件
  • 筛选网格空态
  • 支持 prefers-reduced-motion
  • 添加卡片时自动获取 logo（Clearbit Logo API）

V2（1-3 个月）

  • 邮件/TG bot "Watch a card" —— 推送费率变动、状态变化、新活动
  • 运营情报发布后台（替代 Supabase dashboard）
  • 评价搜索和筛选
  • 卡片对比视图（"Rain vs RedotPay"）
  • 多维度评分（原 spec 的 6 维度评分回归）
  • 评价上下投票用于质量信号
  • 中文本地化（简中为主）

V3 展望

  • 钱包签名验证（可选，加 "已验证持卡人" badge）
  • 历史状态时间轴（查看过去 6 个月 X 卡何时变红/黄）
  • 费率历史图（可视化费率演变）
  • 可嵌入状态 widget
  • 移动 App
  • 用户提交情报（含审核）
  • 卡方官方响应机制（"RedotPay 团队官方回复"）

设计决策日志

决策理由
深色主题原生（非浅色带切换）加密用户偏好深色。品牌更清晰。少一种模式维护。
渲染 CC 视觉（非 logo）todey.xyz 用卡实物图（授权成本高）。CSS 渲染的卡避开法律风险且视觉独特。
匿名提交（无 auth）低门槛 > 高质量。质量问题靠 confirm/deny 而非 auth 解决。
运营专属情报创作冷启动期用户不会产出情报。团队扮演编辑部。
单一评分（非 6 维度）转化优先于分类学。多维度在 V2 有数据量时回归。
Slash 提交后不自动跳转Slash 提交就是增长事件。专门的分享页才能抓住这个价值。
不接受用户提交情报无声誉系统时动机不足。V2 功能。
保留 "Slash" 命名（非 "Report"）"Slash" 独特、产品定义性强、/ 符号本身就是品牌。

风险与缓解

风险严重度缓解
竞品刷对手的假 slash 报告中限流 + confirm/deny 加权（V1.1）
卡方因负面报告发律师函低Footer 明确 "Independent, no affiliation"。UGC 类似 Section 230 保护。
情报准确性 —— 活动静默过期中过期日期字段自动隐藏过期活动。运营每周审。
没用户高创始人 X 运营。SEO 针对 "X card review" 词。交叉发布到相关 subreddit。
状态阈值过敏感（假警报）中数据积累后调优。起步保守（15 = 红）。
